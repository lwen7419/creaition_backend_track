## Project Overview

An Intelligent Task Management System built for an internship coding challenge (AI/LLM Developer Track). The project **started on the Backend track and pivoted mid-build to the AI/LLM track** — this history is intentionally preserved in the git log. See "Pivot Context" below.

The core deliverable is a task management API where tasks can also be created and enriched using LLM-powered features (natural language extraction, tag suggestion, semantic search, task breakdown).

## Tech Stack

- **Language:** Python 3.11+
- **API framework:** FastAPI
- **ORM / DB:** SQLAlchemy + SQLite
- **Validation:** Pydantic
- **Vector store:** ChromaDB (for embeddings / semantic search)
- **LLM:** Anthropic API (via `anthropic` Python SDK), key loaded from `.env`, never hardcoded or committed
- **Testing:** pytest + httpx `TestClient`
- **Env management:** python-dotenv, `.env.example` committed, `.env` gitignored

## Data Model

`Task`:
- `id` — unique identifier (int/UUID)
- `title` — string, required
- `description` — string, optional
- `status` — enum: `pending`, `in_progress`, `completed`
- `priority` — enum: `low`, `medium`, `high`
- `created_at` / `updated_at` — timestamps, auto-managed
- `tags` — array of strings, optional
- `parent_task_id` — nullable FK, used for subtasks generated by the breakdown feature

## Core Requirements (do these first, keep lean — not the focus track)

- Standard CRUD endpoints: `POST /tasks`, `GET /tasks/{id}`, `GET /tasks`, `PATCH /tasks/{id}`, `DELETE /tasks/{id}`
- Input validation on all writes (required title, valid enum values)
- Consistent error handling: 404 for missing resources, 422 for validation errors, JSON error body with a clear `detail` message

## AI Features (the actual focus of this build)

1. **Natural language task creation** — `POST /tasks/from-text`. Use Anthropic tool use / function calling to extract `title`, `description`, `due_date`, `priority` from raw text. Pass the current date into the prompt so relative dates ("tomorrow", "next Friday") resolve correctly. Do not do naive regex/string parsing for this. Demonstrate structured LLM extraction.
2. **Tag suggestion** — LLM suggests 2-4 tags from title/description. Return as *suggestions*, don't silently auto-apply over user-provided tags.
3. **Priority recommendation** — either prompt-based or a small sklearn classifier trained on a hand-built example set. Document the choice and trade-off in `docs/DESIGN.md`.
4. **Semantic search** — `GET /tasks/search?q=`. Embed `title + description` into ChromaDB on task create/update; keep vectors in sync on task delete (don't leave stale embeddings). Query embeds the search string and does similarity search.
5. **Task breakdown** — `POST /tasks/{id}/breakdown`. LLM splits a task into 3-6 subtasks, persisted as new `Task` rows with `parent_task_id` set.

Only build 2 of the "AI Assistant Features" list from the challenge beyond extraction/tagging — semantic search and task breakdown are the two selected here. Don't add summarization or similar-task-detection unless explicitly asked.

## Coding Conventions

- Type hints on all function signatures
- Pydantic schemas for every request/response body — no raw dicts passed across layers
- Business logic lives in `services/`, not in route handlers — routes should stay thin
- No bare `except:` — catch specific exceptions, return meaningful HTTP error codes
- Environment variables accessed only through a single `config.py` / settings module, not scattered `os.getenv()` calls
- Keep LLM prompts in a dedicated `prompts.py` or clearly marked constants, not inline strings scattered through service functions

## Testing

- Mock all Anthropic API and ChromaDB calls in tests — tests must run without network access or a live API key
- Cover: CRUD happy paths, validation errors, `/tasks/from-text` with ambiguous input, `/tasks/search` returning the expected top result for an obvious query

## Git / Commit Conventions

- Small, meaningful commits — not one giant commit
- Do not force-push or rewrite history to hide the backend→AI pivot; it's part of the story told in the README
- Commit messages describe *what* and briefly *why*, e.g. `Add semantic search via ChromaDB embeddings`

## Pivot Context (read before touching backend-track leftovers)

Earlier commits in this repo implement Backend-track features (filtering/sorting/pagination, task dependency graph). The project pivoted to the AI/LLM track partway through. When working in this repo:
- Don't delete the earlier backend commits/history
- The dependency-graph and advanced filtering code may be partially built or unused — check before assuming it's wired into current routes
- If asked to "finish the project," prioritize the AI Features section above; backend-track extras are not required deliverables anymore

## What NOT to do

- Don't commit `.env` or any API key
- Don't hardcode the Anthropic API key as a string anywhere, including in tests
- Don't add authentication/JWT unless explicitly requested — out of scope for this challenge
- Don't over-engineer the dependency/filtering backend code further — that track was deprioritized
- Don't reformat or restyle files unrelated to the current task
